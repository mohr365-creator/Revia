# Revia AOS — Software Requirements Specification (AOS-SRS)

Document: REV-AOS-SRS-001 rev B (baseline)
Parent: System requirements REV-SYS-FCS-001 (per ARP4754B allocation)

Conventions: "shall" requirements only; each requirement has a unique,
immutable ID. Source tags `@satisfies <ID>` and test tags
`@verifies <ID>` provide bidirectional traceability
(see `trace-matrix.md`, generated by `tools/trace/gen_trace.py`).

## 1. Kernel high-level requirements (AOS-HLR)

### Foundations
- **AOS-HLR-001** — All kernel interfaces shall use fixed-width integer
  types; no interface shall depend on platform-defined type widths.
- **AOS-HLR-002** — Every kernel service shall return an explicit
  status code enumerating all failure modes of that service.
- **AOS-HLR-003** — The kernel shall perform no dynamic memory
  allocation; all storage shall be statically sized from configured
  limits (`aos_limits.h`).

### Space partitioning
- **AOS-HLR-010** — The kernel shall confine each partition's writes to
  its statically configured memory region (hardware-enforced on flight
  targets via `aos_hal_protect_select`).
- **AOS-HLR-011** — Each partition shall be in exactly one mode of
  {IDLE, COLD_START, WARM_START, NORMAL}; a partition shall be able to
  change only its own mode, and only via the legal transitions
  (COLD/WARM_START→NORMAL, NORMAL→IDLE); the health monitor may
  command any transition.

### Communication
- **AOS-HLR-012** — Sampling ports shall deliver the most recent
  message with an explicit freshness indication derived from the
  configured refresh period; reads before any write shall be
  distinguishable from valid data.
- **AOS-HLR-013** — Queuing ports shall deliver messages in FIFO order
  from bounded storage; when full, new messages shall be rejected with
  an error and queued data shall never be overwritten.
- **AOS-HLR-014** — The kernel shall reject any port operation by a
  partition that does not own the port, and any operation inconsistent
  with the port's configured kind or direction.

### Time partitioning
- **AOS-HLR-015** — The kernel shall schedule partitions per a fixed
  cyclic table of windows within a fixed major frame, configured before
  start and immutable at runtime; every window shall execute in order
  every frame.
- **AOS-HLR-016** — The kernel shall detect any partition exceeding its
  window budget and any major-frame overrun, and shall report each to
  the health monitor (partition-level and module-level respectively).

### Hardware abstraction
- **AOS-HLR-020** — The kernel shall access hardware exclusively
  through the HAL interface (`aos_hal.h`).
- **AOS-HLR-021** — The HAL time base shall be monotonic
  non-decreasing for the duration of module operation.
- **AOS-HLR-022** — Every partition window shall execute under an
  armed watchdog whose budget equals the window duration.

### Health monitoring
- **AOS-HLR-030** — The kernel shall apply the statically configured
  recovery action (LOG, IDLE, WARM_RESTART, COLD_RESTART, SAFE_STATE)
  for each (partition, error) pair; a partition shall not be able to
  raise errors against another partition.
- **AOS-HLR-031** — On SAFE_STATE the kernel shall command the module
  safe state via the HAL; a module with an invalid or absent
  configuration shall never enter NORMAL operation.
- **AOS-HLR-032** — The kernel shall timestamp and record every
  health-monitor event in a bounded log that preserves the earliest
  events on saturation and counts losses; the log shall be readable
  only by the configured maintenance partition.

### Lifecycle
- **AOS-HLR-040** — Before scheduling any partition, the kernel shall
  validate the entire module configuration (limits, name uniqueness,
  window ordering/fit, channel routing and message-size consistency,
  entry points, region definitions) and shall reject operation on any
  inconsistency.
- **AOS-HLR-041** — Cold start shall zeroize all partition regions and
  kernel state so module behaviour after restart is independent of
  prior history.

## 2. Hosted-application requirements (AOS-SRS, R-100 baseline)

- **AOS-SRS-100** — The IOM shall publish validated air data and
  inceptor commands each frame.
- **AOS-SRS-101** — Inter-partition messages shall be fixed-size typed
  structures defined by the ICD (REV-ICD-001).
- **AOS-SRS-102** — The IOM shall mark source data invalid whenever the
  underlying acquisition has failed.
- **AOS-SRS-110** — In normal law the FCS shall command the elevator to
  track the commanded pitch rate (positive command → trailing-edge-up).
- **AOS-SRS-111** — The FCS shall limit elevator commands to ±25°
  under all input conditions.
- **AOS-SRS-112** — The FCS shall limit elevator rate of motion to
  40°/s under all input conditions.
- **AOS-SRS-113** — Above 12° angle of attack the FCS shall command
  nose-down proportional to the excursion, overriding nose-up demands,
  while preserving pilot nose-down authority.
- **AOS-SRS-114** — On loss or staleness of required air data the FCS
  shall revert to direct law within one frame, transient-free, and
  recover to normal law when valid data returns.
- **AOS-SRS-115** — The FCS integrator shall be clamped (anti-windup)
  and shall not integrate against active alpha protection.
- **AOS-SRS-116** — On loss of the pilot command path the FCS shall
  command surfaces to neutral (failsafe) within one frame.
- **AOS-SRS-120** — The DISP shall annunciate every flight-control law
  change (caution for DIRECT, warning for FAILSAFE, advisory for
  NORMAL recovery) and forward annunciations to the maintenance
  recorder.
- **AOS-SRS-121** — The DISP shall annunciate stale flight-control
  data exactly once per staleness episode.
- **AOS-SRS-130** — The FMS shall publish a navigation solution
  (position, groundspeed, track, active waypoint) each frame.
- **AOS-SRS-140** — The MAINT partition shall collect EICAS messages
  and health-monitor events with bounded per-window processing, for
  Part 121 continuing-airworthiness reporting (§121.373).
- **AOS-SRS-150** — The R-100 baseline module shall host IOM, FCS,
  DISP, FMS, and MAINT in a 50 ms major frame per the configured
  schedule, with all partitions reaching NORMAL mode within two frames
  of cold start.
